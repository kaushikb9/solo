solo — end-to-end walkthrough

A personal thinking companion. Telegram captures raw thoughts → SQLite stores them → an LLM classifier runs lazily when /top3 is invoked → ranked priorities come back. 90% deterministic spine; the small agent surface only shows up in V1.

V0 scope slices 1–3 shipped slice 4 in flight Last regen: 2026-05-23

1. What solo is

One process, one DB, one bot. Telegram is the only surface; SQLite is the only store. Every LLM call routes through a single observable wrapper that writes a trace row for cost & debug visibility. Classification is lazy: thoughts sit unprocessed in entries until /top3 wakes the classifier.

Design tension this resolves: agent frameworks are overkill for a system that’s 90% deterministic. solo hand-rolls the small bits that matter (an LLM client wrapper, a trace table, prompts-as-files) and skips the runtime dependencies (no OpenClaw / Hermes / Pi). See docs/architecture.md §2–3.

Invariants:

  • Capture must never fail. The Telegram handler swallows everything; better to drop a reply than to lose a thought.
  • Pull, not push. No daily digests, no nudges. User asks, system responds.
  • Silent by default. Capture replies captured and nothing else.
  • All LLM calls go through LLMClient. Importing openai anywhere else is a convention violation.

4. Slice 1 — Capture

Telegram → SQLite done

Goal: get raw thoughts off the user’s phone and into a durable store, with the capture-must-never-fail invariant.

Long-polling bot, no webhooks. handle_message is wrapped in a try/except that logs & drops on any failure — the user’s thought is never lost to a stack trace. Optional allow-list (SOLO_ALLOWED_CHATS) filters by Telegram chat id.

5. Slice 2 — LLMClient + trace

Single observable entry point for every LLM call done

Goal: one wrapper for OpenRouter calls; every call writes a trace row with tokens, cost, and latency.

Routes through the openai SDK pointed at OpenRouter’s OpenAI-compatible endpoint. Two methods: chat() for plain text and structured() for Pydantic-shaped output via response_format=schema. structured() renders a named prompt from src/solo/prompts/<name>.md. Both methods write a trace row on success and on failure — observability includes errors.

Pricing is a static MODEL_PRICING dict in llm.py; verified against openrouter.ai/models at write time.

#   ts = now()

6. Slice 3 — Lazy classifier

Turn a raw entry into (kind, summary, priority) done

Goal: the smallest correct & idempotent classifier; lazy (runs only when something needs it); never raises.

classify_pending(conn, llm, *, model, limit=50, max_attempts=3):

  1. Reads unclassified rows in created_at ASC order, skipping rows with classification_attempts ≥ max_attempts.
  2. For each row, calls llm.structured("classifier", ClassifyResult, vars={"entry_text": ...}).
  3. On success → apply_classification() writes kind, summary (truncated to 120 chars), priority, and flips classified=1 — but only if the row is still unclassified (race-safe via WHERE classified=0).
  4. On exception → record_classification_failure() bumps the attempt counter; the loop moves on; bounded retry guarantees a poison entry doesn’t block the queue forever.

The function returns the count of rows successfully classified in this call. It is the only function in solo that must never raise on the hot path — it inherits the capture invariant by extension.

Schema (Pydantic)

class ClassifyResult(BaseModel):
    kind:     Literal["idea", "soft_task", "hard_task", "note"]
    summary:  str = Field(min_length=1, max_length=200)
    priority: Literal["low", "medium", "high"]

Prompt (excerpt from src/solo/prompts/classifier.md)

You classify a single thought captured in a personal thinking log.
Categories (kind):
  - idea       — a thought, hypothesis, or open question to explore
  - soft_task  — vague work needing thinking before doing
  - hard_task  — concrete, executable, fits in Apple Reminders

Priority: high / medium / low
summary: one short line (≤ 120 chars) in the user's voice.

7. Slice 4 — /top3 + /log

Surface the value: ranked priorities and a grouped log next

Goal: close the loop. The classifier output has no consumer yet; this slice makes it useful.

Two Telegram commands on top of what’s already shipped:

  • /top3 — (a) call classify_pending first so any backlog drains, then (b) rank by (priority desc, created_at desc) filtered to soft_task & idea, take the top 3.
  • /log — print recent entries grouped by kind; shape TBD in brainstorming.

Open design questions for the brainstorm:

  • Heuristic ranking vs an LLM scoring pass (architecture.md hints at "single LLM scoring pass" as the V0 default).
  • Should /top3 wait synchronously for the backlog to drain, or return what’s already classified and kick off classification in the background?
  • How many entries does /log show? Time-window or count?
  • What does the Telegram reply look like — Markdown, plain text, message-per-item?

Connection & migration; CRUD helpers. WAL mode, sqlite3.Row factory. Migration is idempotent via per-column PRAGMA table_info check.

ClassifyResult schema; classify_pending(conn, llm, ...). Sequential. Bounded retry. Never raises.

Long-polling Telegram entry point. handle_message wraps capture in try/except — never lets a thought die to a stack trace.

10. Architecture decisions

Non-trivial implementation choices that don’t fit in docs/architecture.md. Each is ~250 words, recording the why.

ADR-0001
Trace write timing
Trace row goes in on every call — success and error paths both. Failures are observable.
ADR-0002
Three-module split
llm.py / prompts.py / trace.py instead of one file. Eval harness will want prompts.py standalone.
ADR-0003
Classifier output on entries
Columns on the row, not a side table. Smaller surface; one query for /top3; race-safe via WHERE classified=0.
ADR-0004
LLMClient connection lifecycle
Fresh SQLite connection per trace write. Trade ~2ms/call for zero lifecycle complexity. Revisit when /top3 latency budget bites.
